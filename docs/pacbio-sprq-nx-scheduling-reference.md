# PacBio SPRQ-Nx / Revio scheduling reference

Distilled from PacBio's *Technical overview: Revio system v13.5 + SPRQ-Nx chemistry and SMRT Link v26.1* (PN 103-849-300 Rev 01, May 2026 — PacBio confidential vendor material; the PDF itself is not checked into this repo). This file captures the engineering-relevant facts only, mapped onto this app's actual entities, for whoever next touches cell-reuse or scheduling logic.

Page numbers below refer to that source deck.

## Vocabulary map

| PacBio term | This app's term | Notes |
|---|---|---|
| Acquisition (one load-and-sequence event) | `CellUse` | one sample, one cell, one well, one run — `models/schedule.py` |
| Multi-use SMRT Cell | `Cell` | `code`, `max_uses`, `status`, `first_use_started_at`, `window_breached` — `models/cell.py` |
| "Use 1 / 2 / 3" | consumed `CellUse` rows for a `Cell` | derived live in `derive_cell_state()`, never hand-entered — `services/cell_service.py` |
| SMRT Cell tray (4 cells) | `CellTray` | `models/cell_tray.py` — pinned to one instrument for life, same invariant as a used `Cell`. `Cell.tray_id`/`tray_position` (1-4) link each cell to its tray; both nullable for cells created before this feature or via `bootstrap_cell()`. See "Tray-of-4 eager population" below. Cost/KPI-grouping still doesn't exist — the original `engine/kpis.py` port from `revio-nx-planner.html` was removed as dead code; see the "Numbers" section below if reintroducing it |
| One instrument, one calendar day | `RunBatch` | unique on `(instrument_id, run_date)` |
| A run's timing/status, up to 8 wells (both tray-loading positions) | `Cycle` (DB model, `models/schedule.py`) | 1:1 with `RunBatch`. Don't confuse with `engine/types.py`'s internal `Cycle` dataclass — that's a private, never-persisted scheduling-batch type from the porting of `revio-nx-planner.html` |
| Expired cell, auto-discarded, unusable | `Cell.status == "window_expired"` | sticky once flagged |
| No acquisitions left on a cell | `Cell.status == "exhausted"` | derived, `remaining <= 0` |

## Hard constraints from the instrument — already modeled correctly

1. **Max 3 uses per multi-use SMRT Cell.** PacBio: "Each multi-use SMRT Cell supports up to 3 acquisitions (uses)" (p.10, p.25). App: `Cell.max_uses` defaults to 3, capped `Literal[1, 2, 3]` (`schemas/run.py:58,71`); consumption is always derived live from non-cancelled `CellUse` rows in `derive_cell_state()` (`services/cell_service.py:44-55`), never re-entered by hand — this mirrors the instrument's own auto-tracking rather than trusting operator bookkeeping.

2. **The 108-hour window is one absolute deadline from Use 1, not a per-use timer.** PacBio: "Once removed from cell tray for 1st use, the 3rd use (Use 3) must start within 108 hours (= 4.5 days)"; the countdown "starts when SMRT Cell is first removed from the cell tray for 1st use" (p.25, p.28) — it does **not** reset at Use 2. App: `CELL_LIFETIME_H = 108` (`engine/constants.py:8`), checked as elapsed hours from `Cell.first_use_started_at` in `recompute_status()` (`services/cell_service.py:30-33`). This is the right semantics — resist any "fix" that turns it into a per-use reset.

3. **Expired or exhausted cells are terminal — the instrument allows no override.** PacBio: "NO option to use expired multi-use SMRT Cells for a sequencing run" (p.25, pp.28-30) — enforced by the instrument itself, not a soft warning. App: `remaining <= 0` → `status = "exhausted"`; `window_breached` → `status = "window_expired"` (`services/cell_service.py:36-39`); `place_sample()` raises 409 when a cell has no remaining uses (`services/placement_service.py:125-128`). **Asymmetry to note:** placement does *not* currently 409 on `window_breached` the same way it does on exhaustion — window state today only surfaces as a scheduling-time warning (`window_flags`). See Gaps below.

4. **A cell, once broken out of its tray, never goes back — reuse is strictly sequential.** PacBio: "Once removed from tray, a SMRT Cell is never returned to the tray" (p.25). Starting Use 1 commits a cell to the 108h clock with no undo. App's requirement that a cell's next use fall on a strictly later date (no same-day or out-of-order reuse) is consistent with this — don't add a path that lets a scheduled use be reordered ahead of an earlier one on the same cell.

5. **Reuse is always prioritized over opening a new cell.** PacBio: the instrument "automatically tracks SMRT Cell uses and prioritizes allocation of samples to used cells before new cells" (p.24), explicitly so operators skip loading a second tray when reuse capacity already covers the remaining samples ("There is no need to load SMRT Cell tray 2," p.24). App: both `pack_cells()` and `fill_slots()` sort candidates `(0 if prior else 1, -len(uses))` (`engine/packing.py:69`, `engine/slot_scheduling.py:25`) — reusable cells always sort before fresh ones, most-used first. Already correct; also a good source of user-facing copy if the UI ever wants to explain *why* it picked particular cells ("N reuse slots available — no new cells needed").

## Tray-of-4 eager population

SMRT Cells physically ship in a tray of 4. The moment any one cell in a tray gets a sample, all 4 are registered as real `Cell` rows together (`services/cell_service.py::open_new_tray()`), not lazily — 3 are left `status="open"`, 0 uses, ready to be picked up later. This is called from both "open a new cell" sites: `placement_service.place_sample()` (mode `"new"`) and `auto_fill_service.auto_fill()`'s persist loop.

- **No shared tray-level expiry clock.** Each cell in a tray keeps its own independent 108h-from-its-own-first-use deadline (rule #2 above) — a tray is a grouping label over 4 independently-clocked cells, not a new deadline concept. Two cells in the same physical tray can start their clocks weeks apart, and the grid's per-slot expiry shading (`StageOut.window_hours_elapsed`) reflects each cell's own clock, never a tray-wide one.
- **The 3 unused siblings surface as preferred reuse candidates automatically, with zero engine changes**, because `load_prior_cells()` (`services/engine_bridge.py:41`) already selects every `Cell.status == "open"` row with no floor on `uses_consumed`, and `pack_cells()`'s sort key already puts every "prior" candidate ahead of opening a fresh one (rule #5 above). This directly implements the vendor's "reuse before new cell" priority *between trays*, not just within one.
- **`current_location()` (`cell_service.py:69`) falls back to the tray's instrument** for a cell with no active uses but a `tray_id` set — otherwise an unused sibling wouldn't know which instrument its physical tray is already sitting on, and would incorrectly be offered as free to land anywhere.
- **`pack_cells()`'s `available_days` cap now also bounds prior cells, not just fresh ones** (`engine/packing.py`) — without this, a single-day batch could plan all 3 of an unused sibling's remaining uses onto it, when only 1 could ever actually be placed that day (a cell can only be reused once per calendar day), stranding the other 2 samples as unplaced instead of spreading them across other cells that could take them today. This gap already existed in principle (a real prior cell with `remaining > 1` and only 1 day on offer), but eager tray population is what made it a routine occurrence instead of a rare edge case.
- **A tray-linked cell that goes back to 0 uses (a placement removed/cancelled/re-cancelled) is never deleted as an "orphan placeholder" on its own**, unlike a legacy non-tray cell (`placement_service.py`'s `remove_sample()`/`move_sample()`/`cancel_run()` all key this off `cell.tray_id is None`) — it's a real physical sibling and stays open/reusable *as long as at least one cell in its tray still has real use history*. But if that was the tray's only real use anywhere — every one of its `CELLS_PER_TRAY` cells now sits at 0 uses — the whole `CellTray` and all its `Cell` rows are deleted too (`cell_service.cleanup_tray_if_fully_unused()`, called from the same three sites). Without this, undoing an entire never-confirmed plan (Clear schedule, or deleting every sample in a tray) left a "ghost" tray with reserved cell IDs and no samples lingering in the weekly grid for the rest of the week, with no way to clear it — reported as a bug once eager tray population made an all-zero-use tray common instead of rare. The cutoff for "real use history" is any `CellUse` row at all (including a stopped cell's permanent `cancelled` marker, not just active ones) — only genuinely untouched siblings get swept away.
- **Bootstrap cutover cells (`bootstrap_cell()`) have no tray** — there's no way to know a historical cell's real sibling history, so `tray_id`/`tray_position` stay null for them, same as for any cell created before this feature shipped.
- **Every sibling is pinned to a well the moment its tray opens, via `Cell.home_well`** (migration `c7f3a1e9d5b2`) — `open_new_tray(db, instrument_id, well)` now takes the well the sample is landing in right now, derives which 4-well box of `WELLS` it belongs to (tray 1's `A01-D01` or tray 2's `A02-D02`), and assigns each of the 4 siblings a `home_well` from that box in fixed order. `current_location()` returns `cell.home_well` (not `None`) for a cell with no active uses but a tray set, so an unused sibling surfaces a real `current_well` through `CellOut` just like a used cell would — this also transparently activates the existing "must stay in its own well" guard shared by `place_sample()`/`move_sample()` (`placement_service.py::_resolve_cell_choice()`), no new validation needed.
- **Unused siblings are now shown in the weekly grid too, not just the Cell Picker/Cell Detail page** — this supersedes an earlier call in this feature's first pass (rendering only realised placements, plus one generic "+" per tray box, to avoid inventing a clock-less forward-looking row). `waitingCells.ts::computeUnusedTraySiblingGhost()` renders a never-used sibling as a static, non-fading ghost (no deadline exists yet) in its own `home_well`-derived slot, reusing the same `SchedulerDayCell.tsx` ghost-pinning machinery a real reuse-ghost uses — it shows on every weekday from the tray's `created_at` onward, with no expiry, until it's loaded or discarded.

**Bug found and fixed (reported: Auto Schedule filling a whole week gave one tray's 4 first-use cells non-sequential ids, e.g. 408/413/418/423 instead of 408/409/410/411):** `auto_fill_service.py`'s persist loop called `open_new_tray()` once per *fresh* `PackedCell` that needed a real DB row, keyed only by that packed cell's own synthetic id — with no awareness that several fresh cells landing in different wells of the *same* box (e.g. filling all 4 first-use wells of "tray 1" in one Auto Schedule click) are, physically, one tray box being loaded, not four. Each call opened its own brand-new `CellTray` + 4 `Cell` rows, so 4 samples landing in one box produced 4 separate physical trays (16 `Cell` rows, only 4 ever used) instead of 1 tray (4 rows, all 4 used) — the gap-of-5 id pattern is exactly "1 used cell + 3 abandoned siblings" repeating once per sample instead of once per box. `load_prior_cells()`'s "reuse before new cell" pool already covers this correctly *across* separate Auto Schedule calls (an unused sibling from an earlier call is a valid prior candidate for a later one) — the bug was specific to several *first-time* box-openings happening inside one single call, which that pre-batch snapshot can't see coming. Fixed by caching opened boxes for the lifetime of one `auto_fill()` call (keyed by `(instrument_id, box_start)`, `auto_fill_service.py`'s `opened_boxes`): the first fresh cell touching a box opens it for real; every other fresh cell landing in a different well of that same box within the same call reuses the sibling `open_new_tray()` already created for it, instead of opening another tray. Covered by `tests/integration/test_auto_fill_api.py::test_auto_fill_shares_one_physical_tray_across_fresh_cells_in_the_same_box`.

**A second, compounding bug in the same report ("removing a sample from a cell's 3rd use and rescheduling made a different cell id appear in that same slot"):** unlike manual placement (`place_sample()`/`move_sample()`, which both explicitly check `current_well` before allowing a reuse), the auto-fill engine path (`pack_cells()` + `fill_slots()`) never tracked a candidate cell's *well* at all — `PriorCellInput`/`PackedCell` only ever carried `pinned_instrument_serial`. `fill_slots()` assigned every cell, prior or fresh, whichever well happened to be next-free in a slot (`free_wells[slot].pop(0)`), with zero regard for which well a reused cell was actually, physically sitting in. Combined with the tray-duplication bug above (which left several *decoy* cells with the same `home_well` value scattered across phantom duplicate trays for one box), the packing engine could genuinely pick a different candidate cell than the one actually resident in that well for a later use - surfacing as "the cell id changed" even though no cell was ever renamed; a different real row got selected. Fixed by threading a `pinned_well` field through the same path as `pinned_instrument_serial`: `current_location()`'s well (already computed, previously discarded) now flows `engine_bridge.load_prior_cells()` → `PriorCellInput.pinned_well` → `pack_cells()` → `PackedCell.pinned_well` → `fill_slots()`, which now only lets a pinned cell take its own well (skipping it for a slot rather than relocating it if that exact well isn't free), and pins a fresh cell to its first-assigned well the same way it already pins one to its first-assigned instrument (`slot_scheduling.py`'s existing fix above). `auto_fill_service.py`'s persist loop also now refuses to let its own cancelled-marker well-collision fallback (`_resolve_well`) relocate an already-real (prior) cell to a different well, since that fallback exists for a fresh cell's not-yet-committed placement, not for a physical cell that can't move. Covered by `tests/unit/test_slot_scheduling.py::test_fill_slots_reused_cell_only_takes_its_own_pinned_well`, `test_fill_slots_strands_pinned_cell_when_its_well_is_taken`, `test_fill_slots_pins_a_fresh_cell_to_its_first_assigned_well`, and `tests/unit/test_packing.py::test_pack_carries_pinned_well_through_from_prior_cell_input`.

**Bug found and fixed (reported: clearing the schedule, or deleting every sample in a tray, left that tray's cell IDs behind as a "ghost tray" for the rest of the week even with no samples in it):** two independent bugs, both stemming from "eager tray population" only ever handling the *opening* half of a tray's lifecycle, never the closing half.

1. Backend — `remove_sample()`/`move_sample()`/`cancel_run()` (`placement_service.py`) already deleted a *non*-tray cell once it hit 0 uses, but for a tray-linked cell always just left it "open, 0/3" forever (see the bullet above), with no check for whether the whole tray had gone empty. Fixed by `cell_service.cleanup_tray_if_fully_unused()`: once a use-removal leaves a tray-linked cell at 0 uses, it locks the `CellTray` row (`FOR UPDATE` — serializes concurrent cleanup checks for two sibling cells removed in the same "Clear schedule" `Promise.all` batch) and checks every sibling; only once *all* `CELLS_PER_TRAY` cells have zero `CellUse` rows of any kind does it delete the tray and all its cells. Covered by the updated assertions in `tests/integration/test_placement_api.py::test_remove_sample_reverts_to_backlog_and_cleans_up_emptied_run`/`test_cancel_run_reverts_all_samples_and_deletes_run`/`test_cancel_run_preserves_a_cancelled_stopped_cell_marker`.
2. Frontend — separately, `SchedulerDayCell.tsx`'s `if (!locked) { ... }` gate around building `ghostBySlot` predates `computeUnusedTraySiblingGhost()` (it was written back when `waitingCells` only ever held reuse ghosts, for which suppressing the droppable "place it here" affordance once a run locks is correct) and was never revisited when the unused-sibling ghost was added later — so locking a run blanked out *all* ghosts, including genuinely populated-but-unfilled tray siblings that have nothing to do with accepting a new placement. Fixed by only excluding reuse ghosts (`!ghost.unused`) once locked, and by `SchedulerSlot.tsx`'s `!stage && locked` branch now forwarding the `ghost` prop through to `SchedulerSlotView` (it previously dropped it unconditionally) so the still-populated sibling actually renders, non-interactively, instead of disappearing.

**Bug found and fixed (reported: a tray with 3 real placements still showed a bare "+" for its never-yet-used 4th cell, instead of that cell's own ready-to-load ghost):** `SchedulePage.tsx`'s three "every cell system-wide" reads that ghosts depend on (`waitingCellsQuery` for reuse/unused-sibling ghosts, `blockedCellsQuery` for stopped wells, `terminalCellsQuery` for exhausted/window_expired/retired) — plus `OpenTraysAccordion`'s open-trays query and `useCompatibleCells` — all called `cellsApi.list({..., page_size: 200})` and used only that first page. `GET /api/cells` (`api/cells.py:78`) sorts `created_at DESC` and slices `[start:start+page_size]` with no client-side follow-up, so once a status class held more than 200 matching cells lab-wide, the *oldest* ones silently fell off page 1 — including a tray's never-used sibling, which ages further down the list every time any newer tray opens anywhere in the system. A tray's *used* siblings never showed this symptom because they render from the separate, unpaginated `cyclesQuery` (real stages), not from this capped cells list — only the ghost-only, never-placed cell depended on it, which is exactly why the symptom looked like "3 real cells fine, 1 phantom cell missing" rather than a whole tray disappearing. Reproduced locally by seeding enough extra open cells to push a real tray past the 200 cutoff and confirming its unused siblings vanished from the grid, then reappeared once fixed. Fixed by `cellsApi.listAll()` (`api/cells.ts`), which follows `page`/`total` until every matching cell is collected, used by all 5 call sites above; `CellsPage`'s own browse/search UI deliberately keeps real `page`/`page_size` (it's a user-facing paged list, not a "give me everything" read). Covered by `src/api/cells.test.ts`.

**Bug found and fixed (reported: "Change cell → Use a new cell" on a placement inside an already-fully-populated tray box put a different physical cell's barcode in that well, even though the tray's other 3 wells still had real, active placements that week):** `change_cell()` (`placement_service.py`, mode `"new"`) called `open_new_tray(db, instrument_id, cell_use.well, exclude_tray_id=old_cell.tray_id)`, deliberately excluding the *entire* old tray from `open_new_tray()`'s box-collision check on the assumption that the old tray was being fully vacated as part of the swap and `cleanup_tray_if_fully_unused()` would delete it right after. That assumption silently broke whenever the old tray's *other* wells (siblings) still had real, active placements: the exclusion blinded the collision check to the whole tray, not just the one cell actually being replaced, so a second `CellTray` + 4 more `Cell` rows got minted directly on top of an already-occupied box — two `CellTray` rows both claiming the same instrument+box, since the old tray survived too (its still-live siblings correctly blocked `cleanup_tray_if_fully_unused`). No existing test ever populated a second well in the same box before exercising mode `"new"`, so this was never caught. Fixed by removing `change_cell()` entirely rather than computing "is the rest of this tray actually fully vacated first": there's no real-world scenario for keeping a sample in its exact slot while swapping in a brand-new physical cell — a tray is a real object sitting in one instrument bay, so the only way a slot's cell changes is for its whole physical tray to be reloaded, which already surfaces correctly as an ordinary fresh placement once a box has gone genuinely empty (ordinary `place_sample()`, unaffected by this bug since it never passed an exclusion). `open_new_tray()`'s `exclude_tray_id` parameter (and `_resolve_cell_choice()`'s pass-through of it) was removed along with it — `change_cell()` was the *only* caller that ever passed a non-`None` value; `_move_sample_to_new_cell()` never did, so the box-collision check is now unconditional for every remaining caller, not just hardened for this one. On the frontend, the slot detail popover's "Change cell" and "Unschedule" actions were removed (`SlotDetailPopover.tsx`) in favor of the grid's drag-and-drop, which already implements the correct model end to end: a cell is physically fixed to its tray/well position for life, so dragging a sample reassigns which cell it uses (adopting whichever cell already lives at the destination, per the "must stay in its own well" guard above) rather than ever creating or replacing one; dragging a placed sample off the grid already called the same `remove_sample()` endpoint the old Unschedule button did, gated by the same lock/cancelled conditions, so removing the button lost no functionality. Covered by `tests/integration/test_placement_api.py::test_place_sample_rejects_new_cell_when_its_tray_box_is_fully_populated_with_real_uses`.

**Bug found and fixed (reported: dragging a placed sample from one tray position to the same tray position on a different day made the sample keep its own original cell instead of adopting whatever cell truly lives in that slot, and the destination's real cell appeared to "move" to the day the sample came from):** `move_sample()` (`placement_service.py`) decided whether to reassign a moved sample to a different cell purely by comparing the destination well against wells recorded on the cell's own *other* `CellUse` rows (`other_uses`). A cell with only one use so far has no `other_uses`, so this check was skipped entirely and the code fell through to an in-place update (`cell_use.cycle = dest_cycle; cell_use.well = well`) that changed which well the physical cell occupied - the literal impossible thing this app's whole tray/cell model exists to prevent. Because eager tray-of-4 population (see above) auto-opens a dedicated sibling cell in every well of a box the moment any one of them gets a sample, and because a box's well can genuinely host a *different* physical tray over time (the old one goes fully terminal, a new one loads in later), a same-well destination almost always already has its own real, currently-open resident cell distinct from whatever the dragged sample arrived with - `other_uses` alone could never see this, since it only looks at the *moved* cell's own history, never at what else is sitting in the destination. Fixed by adding a second, independent check alongside the existing `other_uses` one: before falling through to the in-place update, `move_sample()` now also queries for any *other* `Cell` with `status == "open"` whose tray's instrument matches the destination and whose `home_well` equals the destination well (mirroring `open_new_tray()`'s own box-collision query) - if found, the move is routed through the same `_move_sample_to_new_cell()`/`cell_choice` reassignment path a well-conflict move already used, rather than ever letting a cell's own well field change in place. On the frontend, `CellChoicePicker.tsx`'s `wellConflict` had the identical blind spot (only compared the dragged cell's own `current_well` against the destination, gated on `uses_consumed > 1` so it never even looked at a single-use cell) while separately already computing the destination's true resident/ghost cell (`preselectedCellId`, via the existing per-day-aware `waitingCells.ts::computeGhost`) and discarding it whenever `wellConflict` was false. Fixed by broadening `wellConflict` to also fire whenever `preselectedCellId` differs from the dragged cell, and dropping the `uses_consumed > 1` gate on the existing well comparison. Covered by `tests/integration/test_move_endpoint.py::test_move_to_a_different_slot_same_day_requires_cell_choice_for_the_resident_sibling` and `test_move_onto_a_well_whose_tray_has_since_turned_over`. **Not fixed as part of this bug**, deliberately left as a separate, narrower gap: a cell with no other uses yet can still be moved to a *different instrument* freely (`test_move_across_instruments_allowed_when_it_is_the_cells_only_use`) without updating its `CellTray.instrument_id`/`home_well`, leaving its still-open, unused siblings registered on the old instrument - revisit if this is ever reported in practice.

## App-level invariant: a cell can never move between instruments

This is not sourced from the PacBio deck — it's a scheduling invariant specific to this app's model of a physical cell's lifecycle, worth documenting explicitly because it's easy to assume "packing" logic is free to reassign a reusable cell wherever a slot is open.

**Once a `Cell` has a real (non-cancelled) `CellUse`, every subsequent use of that same cell is confined to the instrument it was first used on — for the lifetime of that cell, across both manual placement and autoschedule ("auto-fill"), regardless of how many instruments are selected/offered.** A cell with no prior use is unpinned and free to land on any offered instrument for its first use; once placed, it's pinned for good.

**This is a claim about the physical `Cell`, not about a planned `Sample`.** A recurring point of confusion (worth stating plainly, since it drove real design questions about drag-and-drop): a sample isn't physically loaded onto anything until its run actually executes — the instrument itself is what moves it from its plate onto a cell, at run time. Before that, a sample sitting in an unexecuted `CellUse` is pure planning state. So "a cell can never move between instruments" does **not** mean a planned sample is stuck wherever it was first scheduled — dragging a placed-but-not-yet-run sample onto a different instrument is just re-planning, and always resolves to a (possibly new) physical cell on the destination instrument, never to relocating the original cell. See `move_sample()`'s cross-instrument reassignment below, which implements exactly this: the sample moves freely; the physical cell it was on stays exactly where it was pinned, untouched, available for whatever other uses it still has.

This is enforced at every layer that can assign a cell to an instrument, not just one gate:
- `services/cell_service.py::current_location()` (`cell_service.py:69-76`) derives a cell's pinned instrument from its most recent active `CellUse` → `Cycle` → `RunBatch` → `Instrument.serial_number`. A `Cell` has no direct FK to `Instrument`; location is always derived, never stored, so there's no stale pin state to fall out of sync. "Most recent" is by the use's actual `run_date` (via `use_run_date()`, `cell_service.py:58-66`), not `CellUse.id` — see the bug note below.
- `services/placement_service.py::place_sample()` (`placement_service.py:164-170`) — manual/drag-drop placement — raises a 409 if a cell's current instrument doesn't match the target instrument (there's no existing placement to reassign here, so a genuine cross-instrument attempt at this specific call is just a bad `cell_choice`, not a move). `move_sample()` (`placement_service.py:438-467`) instead treats a destination-instrument mismatch as a trigger for reassigning the *sample* to a different cell (see below) — never a hard rejection — since a drag-move always has a sample to re-plan onto a fresh cell, unlike a raw `cell_choice`.
- `engine/slot_scheduling.py::fill_slots()` (`slot_scheduling.py:68-69`) — the auto-fill packing loop — skips any slot whose instrument doesn't match `cell.pinned_instrument_serial`, falling the sample to `unplaced` rather than crossing instruments even when a slot elsewhere is free. As of the fix below, it also *sets* `pinned_instrument_serial` (`slot_scheduling.py:95-96`) the first time an until-then-unpinned cell is actually placed, so the pin check above also constrains a fresh cell's 2nd/3rd use within the same batch, not just prior cells loaded from the DB. This restriction is unchanged by the manual-move fix above — one Auto Schedule batch still never spreads a single cell's uses across instruments; only interactive drag-and-drop of one already-placed sample gained the ability to cross instruments.
- `services/auto_fill_service.py::auto_fill()` (`auto_fill_service.py:96-104`) pre-filters the reuse candidate pool before packing: a prior cell pinned to an instrument outside the currently-selected/offered set is excluded entirely, so it can't "claim" a sample via barcode compatibility only to strand that sample once `fill_slots` rejects the cross-instrument placement.

Covered by unit tests in `tests/unit/test_slot_scheduling.py`: `test_fill_slots_respects_cross_instrument_pin_when_a_compatible_slot_exists`, `test_fill_slots_strands_pinned_cell_when_only_a_different_instrument_slot_is_offered`, `test_fill_slots_unpinned_cell_can_use_any_offered_instrument` (confirms fresh cells remain free to land anywhere), `test_fill_slots_pins_a_fresh_cell_to_its_first_assigned_instrument`, and `test_fill_slots_fresh_cell_reuses_stay_on_first_instrument_when_available`; end-to-end via `tests/integration/test_auto_fill_api.py::test_auto_fill_keeps_a_reused_cell_on_one_instrument_when_multiple_are_offered`.

**Bug found and fixed (reported: auto-schedule with every instrument selected put one cell's Use 1/2/3 on three different instruments):** `fill_slots()` only ever *checked* `pinned_instrument_serial`, it never *set* it for a cell with no prior DB use — `PackedCell.pinned_instrument_serial` defaults to `None` for a freshly-created cell (`engine/packing.py`'s "open a new cell" branch never passes it), and nothing updated it as that same cell's uses got assigned across a multi-day, multi-instrument batch. So a fresh cell's Use 1 could land on instrument A, Use 2 on instrument B, Use 3 on instrument C, all within one Auto Schedule click — the "once placed, it's pinned for good" sentence above was aspirational, not enforced, for this one case. Fixed by pinning the cell to its first assignment's instrument at the point of placement (`slot_scheduling.py:95-96`).

A second, compounding bug made the symptom look even more broken ("second use" appearing to be scheduled *before* "first use"): the Use 1/2/3 label (`services/run_serializer.py::_use_number()`) and several `cell_service.py` helpers (`current_location()`, `last_use_run_date()`, `first_use_planned_start_at()`, `serialize_cell_detail()`) all ordered a cell's uses by `CellUse.id` (DB insertion order) under the assumption that insertion order tracks chronological (`run_date`) order. That assumption breaks once a cell's uses span more than one instrument, because `auto_fill_service.py`'s persist loop (deliberately, for an unrelated reason — detecting same-batch instrument locks) inserts `CellUse` rows sorted by `(instrument_serial, run_date)`, not chronologically. Fixed by deriving order from each use's actual `run_date` (new `cell_service.py::use_run_date()` helper, `cell_service.py:58-66`) everywhere a cell's uses need chronological ordering, with `CellUse.id` only as a tie-break. `placement_service.py::move_sample()`'s own pin check (`placement_service.py:352`) got the same fix.

A third, unrelated bug surfaced only after the two fixes above changed exactly when `Cell.cell_uses` first gets lazy-loaded during response serialization: `move_sample()` was moving a `CellUse` to a new `Cycle` by writing the raw FK (`cell_use.cycle_id = dest_cycle.id`) instead of going through the ORM relationship. `Cycle.cell_uses` is `cascade="all, delete-orphan"` (`models/schedule.py`'s `Cycle` model), and a raw FK write bypasses SQLAlchemy's collection-membership tracking that cascade depends on — under the right relationship-loading timing, the unit of work concluded the row had been orphaned from its old `Cycle` and deleted it outright, silently losing the moved placement (empty `stages` in the response). Fixed by assigning through the relationship instead (`cell_use.cycle = dest_cycle`, `placement_service.py:389`), which keeps both sides of the `back_populates` pair in sync and gives the cascade an unambiguous "moved to a new parent" signal instead of an ambiguous "orphaned" one. This class of bug is specific to `Cycle.cell_uses`'s `delete-orphan` cascade; `Cell.cell_uses` (`models/cell.py`) has no such cascade, so a raw `cell_id` FK write is safe there.

**If this ever needs to change** (e.g. to support an operator physically relocating a multi-use cell to a different instrument), it requires a deliberate design decision, not a one-line relaxation: `slot_scheduling.py:68-69,95-96` and `auto_fill_service.py:101-104` would both need to relax together, `placement_service.py:165` would need an explicit override path, and `current_location()`/`pinned_instrument_serial` would need a defined meaning for "reassignment" — today the model has no way to represent a cell moving mid-lifecycle short of falsifying its `CellUse` history. This paragraph is specifically about the *engine* (auto-fill's `fill_slots`/`auto_fill_service`) and about `place_sample`'s raw `cell_choice` path — it is unaffected by the manual drag-and-drop fix below, which only ever reassigns a *sample* to a fresh cell, never relocates a `Cell` row itself.

**Bug found and fixed (reported: dragging a placed sample onto a different instrument was flatly rejected, and — the more serious half — dragging a cell with no other uses yet onto a different instrument silently succeeded by rewriting its existing `CellUse` onto the destination instrument's cycle, corrupting the cell's own derived pin):** this closes the gap the previous bug note above deliberately deferred. `move_sample()` decided whether to reassign a moved sample to a different cell using only two signals: a well mismatch against the cell's *other* uses, and (after the fix above) a different resident cell already sitting in the destination well. Neither signal ever looked at the *destination instrument* itself. A cell with other uses pinning it elsewhere got a hard 409 if the destination instrument differed (`"Cell {code} is already in use on instrument {pinned}; cannot move it to {instrument}"`) — correct in spirit (the physical cell really can't go there) but wrong in response: a sample isn't physically loaded onto anything until its run executes (see the clarifying paragraph above), so this should reassign the sample to a new cell, not refuse outright. Worse, a cell with *no* other uses yet skipped the instrument check entirely (the check was nested inside `if other_uses:`), so the code fell through to the in-place update and silently moved that `CellUse` onto the destination instrument's `Cycle` — the physical cell's `CellTray.instrument_id` never actually changed, so `current_location()` would then derive a pin that disagreed with where the cell's own tray really was. Fixed by computing the cell's pinned instrument unconditionally (falling back to this very use's own run batch when there are no other uses, since that's the only placement that has ever pinned it anywhere) and treating *any* instrument mismatch — with or without other uses — as a trigger for the same `_move_sample_to_new_cell()` reassignment path the well-conflict case already used, never a rejection. `CellChoicePicker.tsx`'s `wellConflict` got the matching fix: two different instruments' grids reuse the same well-label set (e.g. both have an `"A01"`), so a well-string match alone never proved the drop was staying on the same physical cell — `wellConflict` now also fires whenever the drag's source instrument differs from the drop's destination instrument. Covered by `tests/integration/test_move_endpoint.py::test_move_across_instruments_reassigns_to_a_new_cell_when_cell_has_another_use` and `test_move_across_instruments_reassigns_to_a_new_cell_even_as_the_cells_only_use`.

**Bug found and fixed (reported: Auto Schedule on one instrument for a full working week with "Cells per day" set to 4 put 5 uses on one physical cell — one more than the hard 3-use cap):** `fill_slots()` rebuilds `free_wells` fresh for every calendar day, so a well a fresh cell claimed on Monday shows "free" again on Thursday the moment that cell's own pack-assigned quota runs out (e.g. it hit its individual 3-use cap by Wednesday) or it's simply not reused that day. Nothing stopped that "vacated" well from being handed to a *different*, brand-new `PackedCell` for the rest of the batch — each individual `PackedCell` still honored the 3-use cap on its own, but `auto_fill_service.py`'s persist loop resolves a well back to whichever physical `Cell` its box was first opened with (`opened_boxes`, keyed only by `(instrument, box)`, not by which logical packed cell is entitled to a given well), so two unrelated `PackedCell`s landing in the same well (Mon–Wed for the first, Thu–Fri for the second) silently stacked onto the *same* real `Cell` row — 3 + 2 = 5 real `CellUse` rows on one cell. This is specific to `cells_per_day=4`: with tray 2 also open (the default, 8), a full-tray run's lock spills into the next day (see "Instrument load-lock timing" below) often enough that this exact midweek-exhaustion-then-reopen window doesn't line up, which is why it went unnoticed.

Fixed in two layers, since a first pass that only addressed the overuse over-corrected into refusing to ever reload a genuinely spent well within one batch (reported as a follow-up: "auto-schedule now stops after day 3 across the board — day 4 should open a new tray, reused day 5"), which is explicitly legitimate (`open_new_tray()`'s own docstring: "a box whose every cell has gone terminal is not a collision... a brand-new tray can be loaded into it again"):

- `slot_scheduling.py`'s `well_owner` map (`slot_scheduling.py:49-83,121-127,154-156`) is seeded up front with every cell that already has a real, known physical position (prior cells from `engine_bridge.load_prior_cells`), and updated the moment a freshly-opened cell first claims a well. A well already claimed by a *different* `PackedCell` is only offered to a new one once `_well_is_vacated()` confirms the current owner has truly finished — its lifetime total (`total_uses`, already computed by `pack_cells`) has hit `CELL_MAX_USES` *and* every one of its batch-planned uses has actually been placed. A cell `pack_cells` only gave partial work to this batch (e.g. `total_uses` is 1 or 2 — the backlog simply ran out of compatible samples for it, not because it's spent) keeps its well reserved indefinitely, since it may still get reused in a *later*, separate Auto Schedule call and that must land back in the exact same well.
- `auto_fill_service.py`'s persist loop (`auto_fill_service.py:163-172,238-268`) mirrors this at the DB level with a `well_claimant` map, tracked *per well* (not per box — several different fresh `PackedCell`s legitimately share one box at once, one per well). When a different `PackedCell` needs a well the engine has already proven vacated, the loop recomputes the old box's 4 cells' status right then (rather than waiting for this function's own end-of-loop pass) and retires the cached box, so `open_new_tray()`'s own DB-level "is this box still live" collision guard sees them as already exhausted and allows the legitimate reopen — if some sibling in the box genuinely *hasn't* finished (an uneven-quota edge case), that same guard correctly refuses the reopen and the sample is left unplaced rather than corrupting anything. This needs an explicit `db.flush()` right after the `recompute_status()` calls: this project's session runs with `autoflush=False` (`db.py:42`), so without it the status changes stay pending Python-side and the collision guard's own raw query wouldn't see them yet.

`open_new_tray()`'s own DB-level box-collision guard already catches the equivalent mistake *across* two separate Auto Schedule calls (a later call's fresh cell correctly gets refused a well an earlier, still-`open` real cell owns) — the gap was purely within one single call's in-memory plan, before any status had a chance to be recomputed.

One known residual gap, left as a follow-up rather than folded into this fix: `pack_cells()` assigns every fresh cell a flat `min(max_uses, available_days)` depth with no awareness that a *second-generation* cell reloaded mid-batch only has however many days are left after the first generation's wells free up, not the full `available_days` count — so it can over-commit a reloaded cell to 3 planned uses when only 2 days actually remain, leaving a couple of samples honestly reported unplaced rather than repacked onto one more fresh cell that could have taken them. Safe (never over-uses or drops anything silently), just not perfectly optimal; would need pack_cells and fill_slots' day-accounting to be merged to close fully.

Covered by `tests/unit/test_slot_scheduling.py::test_fill_slots_never_hands_a_vacated_well_to_a_different_fresh_cell` and `test_fill_slots_reloads_a_genuinely_exhausted_well_with_a_new_cell`, and `tests/integration/test_auto_fill_api.py::test_auto_fill_never_exceeds_the_hard_three_use_cap_with_cells_per_day_four` and `test_auto_fill_reloads_a_terminal_well_with_a_new_tray_in_the_same_batch`.

## Rules worth mirroring in validation/messaging (not fully enforced yet)

- **Barcode carryover between uses is real, just small.** PacBio quantifies it: "Revio SPRQ-Nx use-to-use barcode carryover level is typically <0.1%" (p.19 footnote) — non-zero, which is exactly why this app's burned-barcode conflict guard exists. `place_sample()` rejects reusing a cell if the new sample shares an already-burned barcode (`services/placement_service.py:129-130`), covered by `test_place_sample_rejects_barcode_conflict_on_existing_cell` (`tests/integration/test_placement_api.py:80`), `test_cell_with_remaining_capacity_is_reused_across_days_and_burned_barcodes_respected` (`tests/integration/test_cell_reuse_across_placements.py:39`), and `test_pack_excludes_prior_cell_when_sample_shares_a_burned_barcode` (`tests/unit/test_packing.py:52`). This is a stricter, zero-tolerance rule than PacBio's own instrument (which tolerates the <0.1% and demultiplexes anyway) — a deliberate safety margin, not a bug; don't loosen it without weighing that tradeoff explicitly. It's a hard block with no user override, by design — the only lever available is which cell the sample lands on, never "proceed anyway".
  - **Frontend gap fixed:** dropping a sample directly onto a specific waiting-cell ghost that turns out to share a burned barcode with it used to fail silently — `CellChoicePicker`'s auto-place logic (`cellChoiceGate.ts::shouldAutoPlace`) only ever checked whether the preselected cell was in the barcode-filtered `compatible` list, so an excluded (clashing) ghost just quietly fell back to opening a brand-new cell instead, with no indication to the user that their intended cell was rejected or why. Fixed by explicitly detecting a barcode clash on the preselected cell (`CellChoicePicker.tsx`'s `preselectedBarcodeClash`, compared against the *raw* open-cells list rather than the pre-filtered `compatible` one) and forcing the picker modal open with a prominent, specific warning (naming the clashing barcode and cell) instead of ever auto-resolving past it silently.

- **Later uses of a cell tend to yield less.** PacBio's own HG002 WGS data across 3 uses of the same cell (p.35) shows Use 3 consistently lower than Use 1/2 (e.g. 143.9 / 144.2 / 119.9 Gb; 147.0 / 148.6 / 124.7 Gb; 148.9 / 154.5 / 136.7 Gb), attributed to lower P1 loading efficiency on later uses — an expected pattern, not an anomaly (footnote, pp.35-36). This app doesn't model per-use yield/coverage at all today (the removed `engine/kpis.py` only ever did cost). If yield estimation is ever added, don't assume flat output across Uses 1-3 of a cell.

- **Sub-5kb amplicon libraries shouldn't go on multi-use cells.** PacBio: "For amplicon libraries <5 kb, using Revio SPRQ sequencing plate with multi-use Revio SMRT Cells may result in reduced P1/.HiFi yield performance for Use 2 and/or Use 3" (p.14) — the recommended path for those is single-use cells instead. `ParsedSample` carries `oplc`/volume but has no insert-size- or library-type-aware validation today — if insert size ever gets captured, this is a concrete rule to enforce (hard block or at least a scheduling warning), not something to infer later from yield data.

## Cadence / batching details consistent with current design

- One `movie_hours` value (12h/24h/30h) per run, never mixed — matches PacBio run design, where one acquisition time applies to every well in a plate/run.
- Up to 8 wells per run across both tray-loading positions, 4 cells per physical SMRT Cell tray (`WELLS` in `engine/constants.py`, 8 entries split into tray-1/tray-2 halves) — matches the Revio's 4-stage-per-tray, 2-tray deck. `STAGES_PER_MACHINE = 4` only ever existed in the pre-redesign `revio-nx-planner.html` prototype (single tray, 4 wells); this app's port replaced it with the 8-well `WELLS` array once the two-tray loading redesign shipped — see "Instrument load-lock timing" below for how a run's lock duration depends on whether tray 2 is loaded too.
- This app's weekday-only run cadence matches the cadence PacBio itself uses in its own multi-use scheduling examples (Mon/Wed/Fri runs — pp.19, 24-25, 32-33), though that reads as a lab-staffing choice layered on top, not an instrument limitation. Worth remembering if this planner is ever used by a lab that runs weekends.
- Reuse cell-prep is modeled as faster than first-use prep (`FIRST_PREP_H = 2`, `REUSE_PREP_H = 0.75`) — qualitatively consistent with PacBio's description of reuse skipping full cell immobilization ("automatic washing, no manual steps," cells "stay in instrument," p.19). Note PacBio's own *quantified* figure is a different comparison: "+45 min added to cell prep time for Use 2 and Use 3 ... compared to cell prep time for single-use SMRT Cells" (p.33 footnote) — i.e. multi-use reuse vs. a single-use cell's one-time prep, not multi-use Use 1 vs. multi-use Use 2/3 on the same cell. Don't assume `FIRST_PREP_H`/`REUSE_PREP_H` are literally sourced from that +45min figure; keep the two comparisons distinct if turnaround timing is ever recalibrated against vendor numbers.

## Instrument load-lock timing

Distilled from three additional PacBio slides (Revio scheduling/utilization examples, not in the numbered deck above) showing per-well acquisition timing and two "high-utilization schedule" Gantt examples. These informed `instrument_lock.py::cycle_lock_until()` - the function that decides when an instrument becomes available to start a brand-new (instrument, day) run.

- **Per-well timing inside one run:** cell prep is ~4h, offset by 2h between wells; PPA (post-primary analysis) starts 28h after loading and recurs every 28h for back-to-back acquisitions (4h prep + 24h sequencing); PPA itself runs ~14h for 4 SMRT cells (a single 2h offset + two 6h units). This app doesn't model per-well PPA timing today - noted here only so a future addition doesn't have to re-derive it.
- **Example high-utilization cadences**, both averaging 24 SMRT cells/week: loading 8 cells per touch point (both trays at once) needs only 3 touch points/week (Mon/Wed/Fri, i.e. loads 48h apart); loading 4 cells per touch point (one tray) needs 6 touch points/week (Mon-Sat, loads 24h apart). The slides label the recurring per-touch-point window where the instrument accepts a new load as the "load window" - everything outside it is the instrument mid-run and unavailable to load.
- **The rule this app implements** (`cycle_lock_until()`): loading only tray 1 (≤4 wells, `WELLS[0:4]`) commits the instrument for `LOCK_BUFFER_HOURS` (6h) only - a short loading/setup window, after which the instrument is free again (to load tray 2 onto the same run, or start a different instrument-day). Loading tray 2 as well (any of `WELLS[4:8]`) commits the instrument to the full movie: locked until `movie_hours` completes, plus another `LOCK_BUFFER_HOURS` (6h) setup for whatever runs next. Both cases derive `lock_until` from the cycle's single `planned_start_at` - this app doesn't record a separate real-world timestamp for when tray 2 specifically went in, so the tray-2 case assumes it was loaded at/near the same time as tray 1.
- **Run design's "Cells per day" (4/8) and "Optimise for: Utilisation" controls directly implement the two example cadences above.** `AutoFillRequest.cells_per_day` (`schemas/run.py`) caps how many of a slot's 8 wells `fill_slots()` will offer per instrument-day (`WELLS[:cells_per_day]`, `engine/slot_scheduling.py`) - set to 4, auto-fill only ever proposes tray 1, matching the "4 cells per touch point" cadence; left at 8 (default), it can fill both trays, matching "8 cells per touch point". The `"utilisation"` objective (`engine/packing.py::pack_cells`) makes packing itself favor that same full-tray shape: unlike `"fewest"/"balance"/"fastest"` (which only ever tie-break between already-open candidates), it refuses to deepen reuse on any fresh cell until `cells_per_day` distinct fresh cells are open, so a touch point opens enough physically distinct cells to fill its wells before any of them starts a 2nd/3rd use - deepening only kicks in once that width is reached. Both are additive to the existing reuse-before-new-cell priority (rule #5 above), which is unaffected by either setting.
- This lock is explicitly a **planning tool**: it only gates *creating a brand-new* (instrument, day) run (`placement_service.get_or_create_run`) so a user isn't invited to schedule a run the instrument can't have started yet - see the "Gaps" note below on `planned` vs `running`. Adding more wells to an already-existing run is never blocked by it.

## Gaps / open questions worth a deliberate decision

1. Should `window_breached` block placement (409) the same way exhaustion does, given PacBio treats both as equally hard, uncircumventable stops on the real instrument? Today only exhaustion blocks placement; window breach is scheduling-time-only (`window_flags`).
2. The 108h check applies uniformly regardless of which use is next (2nd or 3rd) — this matches PacBio's stated single-deadline behavior (see above), but if the instrument's actual firmware ever turns out to differentiate by use-index, or the SMRT Link "start within" countdown (p.29) implies something more granular, revisit `recompute_status()`.
3. No insert-size / library-type-aware validation exists — the planner will currently pack a <5kb amplicon sample onto a cell's 3rd use exactly like a WGS sample, which PacBio's own docs flag as a yield risk (see above).
4. Real instrument operators can manually discard *all* on-stage multi-use cells at once, not individually selectable (p.27) — physical instrument state can diverge from anything this planner scheduled, with no partial-recovery path. This app has no instrument integration today, so treat any generated schedule as advisory, not a guarantee of what will actually run.
5. Instrument locking (see "Instrument load-lock timing" above) is deliberately keyed off a `Cycle` merely being `planned`, not `running`/confirmed-loaded - the lock is a planning tool describing when the instrument *will* be busy once the plan is executed, not a report of real-world state. If this app ever integrates with the real instrument, revisit whether an unconfirmed `planned` cycle should still lock the next day the same way a confirmed one does.
